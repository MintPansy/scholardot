## 참여 동기

저는 웹 프론트엔드 개발을 공부해 오면서, **실제로 기획·디자인과 연결되어 배포까지 이어지는 프로젝트**를 해보고 싶다는 생각이 있었습니다. 강의나 개인 연습만으로는 “디자인 시안을 받아 구현해 보는 경험”, “백엔드 API와 협업하는 경험”이 부족했고, 포트폴리오에 담을 만한 **완성된 서비스**가 없었습니다.

스위프 웹을 알게 되었을 때, **PM·디자이너·개발자가 한 팀이 되어 6주 만에 아이데이션부터 출시까지** 진행한다는 점이 가장 끌렸습니다. 단순 학습이 아니라 **실제 서비스로 완성된 결과물**을 만들 수 있고, **완성도 있는 프로젝트를 위한 다양한 인프라**를 스위프에서 제공한다는 설명이 참여 결심에 큰 이유가 되었습니다. 그래서 Paper Dot 팀의 프론트엔드 개발자로 합류하게 되었습니다.

## 가장 인상 깊었던 경험

가장 인상 깊었던 것은 **비동기 번역 처리**를 사용자 경험에 맞게 설계하고 구현한 과정이었습니다. PDF를 업로드하면 백엔드에서 번역이 비동기로 진행되는데, “언제 끝났는지”를 프론트에서 알아야 했습니다. 단순히 한 번 요청하고 기다리기만 하면 사용자는 진행 상황을 알 수 없고, 번역이 끝나기 전에 조회하면 404가 나와 에러처럼 보이는 문제도 있었습니다.

그래서 **폴링 로직**을 도입했습니다. 번역 요청 후 일정 간격(3초)으로 최대 20회까지 결과를 조회하고, 404는 “아직 처리 중”으로 간주해 에러가 아니라 대기 상태로 처리했습니다. 이렇게 하니 사용자는 다른 작업을 하면서도 번역이 완료되면 자연스럽게 결과를 볼 수 있었고, 네트워크나 서버의 일시적인 문제에도 더 안정적으로 대응할 수 있었습니다. “기술적 제약을 UX 관점에서 어떻게 풀 것인가”를 팀과 이야기하고 구현해 본 경험이 많이 기억에 남습니다.

## 배운 것과 성장한 부분

**상태 관리**에서는 Redux의 보일러플레이트 없이 가볍게 쓰고 싶어 **Zustand**를 도입했습니다. 로그인 상태, 문서 ID, 토큰 등을 스토어별로 나누어 약 40줄 수준으로 정리했고, 필요한 컴포넌트만 구독하도록 해서 불필요한 리렌더를 줄일 수 있었습니다.

**PDF.js**를 활용해 페이지별 텍스트를 추출하고, 번역된 문장들을 **원문과 페이지 단위로 매칭**하는 로직도 직접 설계해 봤습니다. 대소문자·특수문자 차이로 매칭이 깨지는 문제는 정규화 함수로 해결하고, 연속 단어 3~5개로 먼저 검색한 뒤 실패 시 1~2개로 폴백하는 방식으로 정확도와 성능 사이의 균형을 맞추었습니다. “데이터 매칭과 정규화”에 대한 감이 많이 생겼습니다.

또 **OAuth(Google, Kakao)** 플로우를 Next.js API Routes와 쿠키, Referer 등으로 처리하면서, 서버와 클라이언트 사이의 인증 상태 동기화를 경험했고, **Content-Type**을 보고 HTML 응답(로그인 페이지 등)과 JSON을 구분해 에러 메시지를 나누는 등 API 연동과 에러 처리에 대한 감각도 키울 수 있었습니다.

## 어려웠던 점과 극복한 방법

**PDF.js Worker** 로드 이슈가 있었습니다. CDN으로 worker를 쓰면 Vercel 배포 환경에서 `Source file not found`가 나왔습니다. 해결 방법은 `node_modules/pdfjs-dist`의 worker 파일을 `public/`으로 복사하고, `GlobalWorkerOptions.workerSrc`를 그 경로로 지정한 뒤 빌드 시 자동 복사되도록 설정한 것이었습니다. “CDN에만 의존하지 말고, 배포 환경에 맞는 로컬 리소스 전략”을 고민하는 계기가 되었습니다.

**번역 완료 시점 감지**도 난제였습니다. 404를 그대로 에러로 처리하면 “번역 실패”로 오해할 수 있어서, “404 = 처리 중”으로 정의하고 폴링으로 주기적으로 조회하는 방식으로 바꿨습니다. 비동기 작업의 상태를 어떻게 나누고, 사용자에게 어떻게 보여 줄지에 대한 설계 경험을 쌓을 수 있었습니다.

## 프로젝트 소감

Paper Dot은 “영어 논문 PDF → 한글 번역 → 원문과 번역문을 함께 읽기”라는 목표를 **실제 서비스**로 만들어 본 첫 경험이었습니다. 단순히 화면만 만드는 것이 아니라, **인증·API·상태·에러 처리·PDF 매칭**까지 프론트엔드에서 고려할 부분을 직접 설계하고 구현해 보면서, “기능 하나를 끝까지 책임지는” 느낌을 받았습니다. 스위프에서 강조하는 **“단순 학습이 아니라 실제 서비스로 완성된 결과물”**을 경험했다는 점이 가장 큰 수확입니다.

## 스위프 참여 소감과 혜택

스위프 웹의 **“6주 만에 웹 개발부터 출시까지, 완성도 있는 프로젝트를 위한 다양한 인프라 제공”**이라는 점을 실제로 체감했습니다. 기한과 단계가 정해져 있어서 “끝까지 완성해야 한다”는 동기가 생겼고, 팀원들과 역할을 나누어 진행하면서 **PM·디자이너·개발자가 한 팀이 되어 실제 현업과 가까운 환경에서 협업**해 보는 경험이 됐습니다.

또 **“아이데이션부터 기획, 디자인, 개발까지 웹 서비스 제작의 전 과정을 직접 경험”**할 수 있어서, 프론트엔드 개발자로서 “기획·디자인과 어떻게 맞춰 나갈지”, “API와 어떻게 협업할지”에 대한 감이 많이 생겼습니다. 스위프 운영진의 적극적인 지원 덕분에 막힐 때마다 방향을 잡기 수월했고, 데모데이와 네트워킹을 통해 다른 팀의 결과물과 인사이트도 공유할 수 있어서 좋았습니다.
